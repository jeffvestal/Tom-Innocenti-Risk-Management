Rerankers — The Precision Layer

Why two stages? Embedding search is fast and scales to millions of documents, but it works by comparing pre-computed vectors and can miss subtle relationships. A reranker reads each candidate alongside the query in full detail and re-scores them. You get speed from embeddings (find 100 candidates in milliseconds) and precision from reranking (surface the best 10).

Reranker v3 Recommended

  • Listwise scoring — processes the query and up to 64 documents simultaneously in a single pass
  • Compares documents against each other, not just against the query. “Which of these is most relevant?” instead of “Is this relevant?”
  • 600M params, 131K token context
  • 100+ languages, outperforms models with 2.5× more parameters
  • Best when: candidates are close in relevance and precision matters

Reranker v2

  • Pointwise scoring — each document scored independently against the query
  • 278M params, 1K token context
  • 100+ languages, 6× faster than v1
  • Unique: function-calling support (match queries to APIs) + code search
  • Best when: high-throughput pipelines where speed matters more than marginal precision
The analogy: Embedding search is scanning a library catalog to find the right shelf. The reranker is a librarian who reads the actual pages and tells you which books best answer your specific question.
Jina AI

Two Things FEs Get Wrong About Rerankers

(1) The first stage doesn’t have to be semantic.
Keyword/BM25 works too. The reranker doesn’t care how you found the candidates — it just needs a list of documents to re-score.
(2) The reranker reads the raw text, not the embeddings.
It re-processes each document alongside the query from scratch. First-stage embeddings (if any) are not reused. Even if your first stage was keyword-only, the reranker still works.
Jina AI

Listwise vs Pointwise Scoring

Why listwise is better
Imagine 5 results all mention “facial recognition in law enforcement.” Pointwise scores each one in isolation — you get 0.85, 0.83, 0.82, 0.81, 0.80. Nearly identical, no way to distinguish the best answer.

Listwise reads all 5 together and asks: “Which of these best answers the specific question?” It can spot that doc #3 discusses the exceptions to the prohibition — which is what you actually asked about — while the others just describe the general rule.
Why pointwise still exists
Listwise is computationally heavier — it has to attend across all candidates simultaneously. Pointwise (v2) is significantly faster, half the parameters (278M vs 600M), and has unique capabilities listwise doesn’t: function-calling support and code search.

When your candidates are clearly different in quality, pointwise is plenty good — and faster.
The analogy
Pointwise = each essay graded independently against a rubric. Works fine when quality varies widely.

Listwise = teacher reads all essays, then ranks them. Essential when they’re close in quality and you need to pick the best one.

Visual Intelligence — CLIP & VLM

CLIP v2 — Cross-Modal Search

Separate text and image encoders trained to produce embeddings in the same vector space. A text query like “red sports car” finds matching images. An image of a car finds matching product descriptions. This is cross-modal retrieval.
  • 865M params, 89 languages
  • 512×512 image resolution (4× v1)
  • Matryoshka: truncate 1,024 → 64 dims with <1% loss
  • Lighter than v4 — choose for speed/cost when you don’t need document understanding

VLM — Vision Language Model

A generative model: it looks at an image and produces text answers to questions about it. Not retrieval — understanding. “What does this chart show?” “What text is in this scanned document?”
  • 2.4B params, 29 languages
  • Best multilingual VQA among open 2B models
  • Strong OCR (778/1000 OCRBench)
  • Lowest hallucination rate (90.3 POPE) in its class
  • Available via Jina API — not on EIS
CLIP vs VLM vs v4: CLIP v2 embeds images for search (lightweight, fast). v4 embeds images + text together for multimodal search (heavier, handles complex visual docs). VLM generates text about images (answers questions, extracts info). They can be used together in a pipeline. Reader and VLM are accessed via the Jina API directly — they are not available through EIS.

What Is EIS?

Historically, running ML models in Elasticsearch meant provisioning dedicated ML nodes, managing GPU resources, scaling allocations, and monitoring model health. That’s a lot of infrastructure just to enable semantic search.

  • Managed ML inference as a service — embeddings, reranking, and LLM chat without deploying any ML nodes
  • No GPU management — Elastic runs the models on their infrastructure
  • Billed per million tokens, not VCU hours or node count
  • Data stays in your cluster — only the text sent for inference is transmitted, discarded immediately after processing
The key idea: Instead of provisioning and scaling ML nodes yourself, EIS handles all of that. You configure an inference endpoint — one API call — and Elasticsearch uses it automatically for ingest, search, and chat. Zero infrastructure to manage.

What Is an Embedding?

When an AI model reads text, it converts it into a list of numbers — a vector.

The key insight: similar meanings produce similar numbers.

A search for “how to fix a leaky faucet” will match a document about “repairing dripping taps” — even though the words are completely different.

This is how semantic search works.
Jina AI

What Is a Context Window?

The maximum text a model can process in one pass.

512 tokens ≈ 1 paragraph.
8K tokens ≈ 6 pages.
32K tokens ≈ 25 pages.

A bigger window doesn’t automatically mean better results — it means more flexibility.

You choose how to chunk your data instead of the model forcing you to.
Jina AI

What Is Matryoshka Representation Learning?

Named after Russian nesting dolls.

The model front-loads the most important information into the first dimensions of the embedding.

Truncate from 1,024 down to 256 dimensions and keep 90%+ of search quality.

Trade a tiny accuracy reduction for major storage savings and faster searches.
In Elasticsearch: Set dimensions when creating a custom inference endpoint. Default is 1,024 (v5-text-small) or 768 (v5-text-nano). Reduce to 256 or 512 for faster search with minimal quality loss. The semantic_text field uses whatever dimensions the endpoint specifies.
Jina AI

What Are LoRA Adapters?

Small, lightweight tuning layers (~5M parameters each) attached to one base model.

The base model already handles retrieval, classification, and clustering on its own — but adapters optimize it for each task.

Select the adapter at query time.

One deployment, multiple specializations.
In Elasticsearch: Set task_type when creating the inference endpoint — options: search, ingest, classification, clustering. Elasticsearch automatically selects the right LoRA adapter at inference time. No manual adapter management needed.

curl "https://r.jina.ai/https://example.com" # → clean Markdown curl "https://s.jina.ai/EU AI Act biometric" # → web search → clean results
Embeddingsapi.jina.ai
Generate embeddings via REST. Supports all Jina embedding models (v3, v5, CLIP). Pass text or images, get vectors back.
curl https://api.jina.ai/v1/embeddings \ -H "Authorization: Bearer <KEY>" \ -d '{ "model": "jina-embeddings-v3", "input": ["Your text here"] }'
Rerankerapi.jina.ai
Pass a query + documents, get them re-scored and re-ordered by relevance. Same v2/v3 models available on EIS.
curl https://api.jina.ai/v1/rerank \ -H "Authorization: Bearer <KEY>" \ -d '{ "model": "jina-reranker-v3", "query": "AI regulation", "documents": ["..."] }'

Jina API — Tips & Integration

MCP Server: Add mcp.jina.ai as an MCP server to access Reader, Embeddings, and Reranker directly from LLMs and AI agents. No custom integration code required.
Getting an API Key: Go to jina.ai → click “How to get my API key?” → key appears in the API Key & Billing tab. Free tier with rate limits; add your key to requests for higher throughput.

The questions you’ll get on the first call. Have answers ready.

Common Customer Questions

“What about latency?”
EIS inference runs on Elastic’s infrastructure, co-located with your cluster. Typical embedding latency is low single-digit milliseconds per document. Reranking adds ~50–100ms for a batch of 50 docs. Compare that to an external API round-trip.
“What about our existing ELSER?”
It keeps working. ELSER is still a supported pre-configured endpoint. For new deployments, recommend v5-text-small. Migration is a reindex operation, not a breaking change.
“We’re self-managed. Can we use this?”
Yes, via Cloud Connect (Stack 9.3+). It bridges your cluster to Elastic Cloud services. Requires Enterprise license or active trial.
“Does data leave the cluster?”
For EIS-managed models (Modes 1 & 2): text is sent to Elastic’s inference infrastructure, processed, and immediately discarded. 0-day retention. Not used for model training.
“Non-English content?”
v5-text-small supports 119+ languages out of the box. No language-specific configuration needed. Cross-lingual search (query in English, find German docs) works automatically.
“Can we fine-tune the models?”
v5 supports LoRA adapters for task-specific tuning (retrieval, classification, clustering, STS). Select the adapter at query time — one model, multiple tasks.

Gotchas & Anti-Patterns

Don’t PUT _inference for pre-configured endpoints.
They start with a dot (.jina-embeddings-v5-text-small) and already exist. Calling PUT on them will error. Just reference the dot-prefixed ID in your mapping.
Don’t recommend Gemini 2.5 Pro for agents.
Tends to time out on complex agent loops. Recommend GPT-4.1 Mini (fastest) or Claude Sonnet 4.5 (best balance) for agent connectors.
VLM ≠ CLIP replacement.
CLIP searches images (cross-modal retrieval). VLM answers questions about images (generative). They’re complementary — don’t position one as replacing the other.
Don’t double-chunk.
semantic_text handles chunking automatically. If a customer is manually chunking AND using semantic_text, they’re chunking twice. Use one or the other.
